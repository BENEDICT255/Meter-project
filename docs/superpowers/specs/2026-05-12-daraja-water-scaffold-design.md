# Daraja Water ‚Äî scaffold design

**Date**: 2026-05-12
**Scope**: Initial scaffold (the "walking skeleton") for a prepaid water meter top-up platform serving Tanzanian customers. v1 is a JSON API only ‚Äî no frontend, no realtime push. Later tasks (2‚Äì5) plug into shaped slots defined here.

## Goals

- A bootable Django + MySQL service that supports the end-to-end buy-token flow: register, add meter, initiate a transaction, receive an HMAC-verified webhook, generate a token, attempt SMS delivery.
- Architectural seams (`TokenStrategy`, `SmsProvider`, `PaymentWebhookView`) shaped to receive the harder implementations in tasks 2‚Äì5 without restructuring.
- Tests for the parts that ship as "done": auth, meter CRUD, transaction initiation, the full webhook flow (signature, idempotency, 404, payload‚Üítoken), `SimpleTokenStrategy`, `ConsoleSmsProvider`.

## Non-goals

- Frontend (Next.js, templates) ‚Äî explicitly cut from scope; API-only.
- WebSockets / Channels / Redis ‚Äî explicitly cut.
- `HmacTokenStrategy`, real SMS providers with retry, `expire_transactions` command, rate limiting, JWT blacklist ‚Äî deferred to their tasks.
- Real payment provider integration (their callback schema, signing algorithm, outbound API) ‚Äî deferred to task 2.

## Stack

- Python 3.12, Django 5, DRF, djangorestframework-simplejwt, mysqlclient, python-dotenv, pytest, pytest-django, requests.
- MySQL (Homebrew) for dev and tests. No SQLite.
- `uv` for env + deps via `pyproject.toml`. `uv sync`, `uv run manage.py ‚Ä¶`.
- No Channels, no Redis, no Celery.

## Data models

### `accounts.User`

`AbstractUser` subclass.
- `phone_number` ‚Äî unique, indexed, E.164 string. `USERNAME_FIELD = "phone_number"`.
- `email` ‚Äî optional.
- The `username` field is removed (`username = None`) so we don't carry a dead column.

### `meters.Meter`

- `id` ‚Äî UUID PK.
- `owner` ‚Äî FK ‚Üí `User`, `on_delete=CASCADE`.
- `meter_number` ‚Äî `CharField`, unique, indexed. The physical meter ID; used as a factor in the token formula. Validated as digits-only (10‚Äì14 chars) so `SimpleTokenStrategy` can use `int(meter_number)` directly.
- `label` ‚Äî optional `CharField` ("Home", "Shop").
- `created_at` ‚Äî `DateTimeField(auto_now_add=True)`.

### `payments.Transaction`

- `id` ‚Äî UUID PK.
- `user` ‚Äî FK ‚Üí `User`.
- `meter` ‚Äî FK ‚Üí `Meter`.
- `amount` ‚Äî `Decimal(max_digits=12, decimal_places=2)`.
- `control_number` ‚Äî `CharField(max_length=12)`, unique, indexed. `"99"` + 10 digits, generated by `payments/control_numbers.py`.
- `status` ‚Äî choices: `pending | paid | expired | failed`. Default `pending`. Indexed.
- `provider_reference` ‚Äî `CharField`, nullable. Set on webhook (real provider's transaction ID).
- `paid_at` ‚Äî `DateTimeField`, nullable.
- `expires_at` ‚Äî `DateTimeField`. Computed at create time as `created_at + TRANSACTION_TTL_MINUTES`.
- `created_at`, `updated_at`.

### `payments.Token`

- `id` ‚Äî UUID PK.
- `transaction` ‚Äî `OneToOneField(Transaction)`. One token per paid transaction; the OneToOne is the structural lock against double-minting.
- `value` ‚Äî `CharField`. The generated numeric token. Length is strategy-dependent.
- `strategy` ‚Äî `CharField` (`"simple" | "hmac"`). Records which strategy minted this token so we can verify with the same algorithm later.
- `delivered_via_sms` ‚Äî `BooleanField`, default `False`.
- `delivered_at` ‚Äî `DateTimeField`, nullable.
- `created_at`.

### Design rationale

- **Phone number as `USERNAME_FIELD`**: Tanzanian customers identify by phone; SMS delivery needs it; making it the login means one less field.
- **UUID PKs throughout**: control numbers and token values are public-facing. Sequential IDs would leak volume and be guessable. UUIDs cost nothing on this scale.
- **One Token per Transaction (`OneToOne`)**: the webhook's idempotency story becomes "if a token already exists for this transaction, return it; don't mint another."
- **`status` enum, not booleans**: a transaction has a lifecycle (pending ‚Üí paid/expired/failed). Booleans for each state lead to invalid combinations.

## API surface

All endpoints under `/api/`, JSON in and out, JWT-authenticated unless noted. DRF routers + `ModelViewSet`s where the resource is CRUD-shaped; plain `APIView` where it isn't (initiate, webhook).

```
GET    /api/auth/me/                                                ‚Üí { user }
```

### Meters (authenticated, owner-scoped)

```
GET    /api/meters/                                                 ‚Üí [{meter}, ‚Ä¶]
POST   /api/meters/           { meter_number, label? }              ‚Üí { meter }
GET    /api/meters/{id}/                                            ‚Üí { meter }
DELETE /api/meters/{id}/                                            ‚Üí 204
```

The viewset's `get_queryset` filters by `request.user`; another user attempting to read or delete a meter sees a 404.

### Transactions (authenticated)

```
POST   /api/transactions/initiate/  { meter_id, amount }            ‚Üí { transaction }
GET    /api/transactions/                                           ‚Üí [{txn}, ‚Ä¶]      # most recent first, user-scoped
GET    /api/transactions/{id}/                                      ‚Üí { transaction, token? }
```

`initiate/` validates `meter_id` belongs to `request.user`, generates a unique control number, creates the `Transaction(status="pending")`, and returns it. Returning the token inline on `GET /api/transactions/{id}/` (when present) is intentional ‚Äî keeps the surface small and the success-page UX simple.

### Webhook (public, HMAC-verified)

```
POST   /api/webhooks/payment/
       headers: X-Signature: sha256=<hex>
       body:    { control_number, amount, provider_reference, status }
       ‚Üí 200 | 401 | 404
```

Field names in the body are placeholders; task 2 will rename them to match the real provider. The verify-then-act pipeline (below) stays.

### Errors

DRF defaults: `{ "detail": "‚Ä¶" }` for top-level errors, per-field arrays for validation errors. No custom error envelope.

These three seams exist precisely so tasks 2‚Äì5 can drop in without restructuring.

### Slot A ‚Äî `payments/token_logic.py`

**Why `nonce`**: the abstract method takes a nonce (the transaction UUID) even though `SimpleTokenStrategy` ignores it. This means task 3's `HmacTokenStrategy` can use it for replay protection without changing the interface.

**Why pass `meter_number` not `meter_id`**: the brief's formula `(amount √ó 1357) + meter_id` is integer arithmetic. The DB PK is a UUID; the physical meter identifier is `meter_number`. Passing `meter_number` keeps the formula faithful and the strategy interface simple.

### Slot C ‚Äî `PaymentWebhookView`

```
1. Read raw request body and X-Signature header.
2. signing.verify_hmac(body, signature, settings.WEBHOOK_HMAC_SECRET)
   ‚Üí invalid ‚Üí return 401 immediately (no DB hit).
3. Parse JSON body; extract control_number.
4. Open transaction.atomic() block:
   txn = Transaction.objects.select_for_update().get(control_number=...)
   ‚Üí DoesNotExist ‚Üí 404
5. If txn.status == "paid":
   token = txn.token         # the existing OneToOne
   ‚Üí return 200 { token }    # idempotent replay
6. Else if payload status indicates paid:
   txn.status = "paid"; txn.paid_at = now; txn.provider_reference = payload[...]
   txn.save()
   strategy = get_strategy()
   value = strategy.generate(amount=txn.amount, meter_number=txn.meter.meter_number, nonce=str(txn.id))
   token = Token.objects.create(transaction=txn, value=value, strategy=strategy.name)
   send_token_sms(token)     # best-effort; failure does NOT roll back the txn
   ‚Üí 200 { token }
7. Else if payload status indicates failed/cancelled:
   txn.status = "failed"; txn.save()
   ‚Üí 200
```

**Idempotency lock**: `select_for_update` inside `transaction.atomic()` serialises concurrent deliveries of the same webhook. Combined with the `OneToOneField`, double-token-minting is structurally impossible.

**SMS failure handling**: `send_token_sms` raising must not undo the transaction's paid state. The token row exists; SMS can be retried. Wrap the SMS call in a try/except that logs and swallows.

**HMAC verifier** (`payments/signing.py`): expects `X-Signature: sha256=<hex>`, uses `hmac.compare_digest` for constant-time comparison, returns bool.

## Testing approach

`pytest` + `pytest-django` against a real MySQL test database (pytest-django creates/destroys it). `conftest.py` at the repo root provides fixtures:

- `user` ‚Äî a saved `User`.
- `api_client` ‚Äî DRF `APIClient` with no auth.
- `authed_client(user)` ‚Äî `APIClient` with `Authorization: Bearer ‚Ä¶` set via SimpleJWT's `RefreshToken.for_user(user).access_token`.
- `meter(user)` ‚Äî a saved `Meter` owned by `user`.

## Manual sanity check (README snippet)

A bash snippet using `curl` and a tiny Python one-liner to compute HMAC:

1. `POST /api/auth/register/` ‚Üí user created
2. `POST /api/auth/login/` ‚Üí grab `access`
3. `POST /api/meters/` with bearer ‚Üí grab `meter_id`
4. `POST /api/transactions/initiate/` ‚Üí grab `control_number` and `transaction.id`
5. Compute `sig = "sha256=" + hmac.new(SECRET, body, sha256).hexdigest()`
6. `POST /api/webhooks/payment/` with `X-Signature: $sig` and `{control_number, amount, provider_reference, status:"paid"}` ‚Üí 200 with token
7. `GET /api/transactions/{id}/` ‚Üí confirms `status=paid`, `token.value` set

Console SMS will log the message to the Django runserver output.

## What this design does NOT lock in

- The real payment provider's webhook payload shape (task 2 reshapes `PaymentWebhookView`'s parsing step; the HMAC + idempotency frame stays).
- The real SMS provider (task 4 adds a class to `sms.py` and updates `_get_provider()`).
- The token strategy beyond `SimpleTokenStrategy` (task 3 adds `HmacTokenStrategy` to `token_logic.py`).

Each later task is additive within an existing module.
